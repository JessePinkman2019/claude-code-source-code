## 第一部分：宋博 — Bootstrapping Agents

### 开篇：什么是 Bootstrapping？

Bootstrapping 来自强化学习（RL）领域，核心意思是：**用自己积累的经验，不断更新自己的认知**，而不依赖外部标注。

> 类比：一个孩子学下棋，没有老师告诉他每一步对不对，只靠最终的输赢结果，自己回头琢磨"哪步棋走对了，哪步走错了"——这就是 bootstrapping。

### 三层框架总览

宋博把 agent 的自我学习划分为三个层次，抽象程度递增：

| 层级 | 名称 | 学习的对象 |
|------|------|-----------|
| Level 1 | Temporal Bootstrapping（时间维度） | 哪个具体动作/词语导致了成败 |
| Level 2 | Representational Bootstrapping（状态表征） | 某个"局面"本身是好是坏 |
| Level 3 | Knowledge Bootstrapping（知识） | 更高层次的规律和技能（尚无具体工作） |

### Level 1：时间维度的自我校正

#### 核心问题：Credit Assignment（功过归因）

AI 完成一个任务会产生很长一串输出（叫做"轨迹"）。任务结束时我们只知道最终成功或失败，但不知道是哪句话、哪个词语起了关键作用。

> 比如 AI 写了一段 500 字的代码，最后跑通了——但到底是哪行代码关键？把功劳平摊给每一行公平吗？

这个"找关键动作"的问题就叫 **Credit Assignment（功过归因）**。

#### 现有方法的做法

- **GRPO**（最常用的一种 RL 算法）：非常粗暴，成功了就把所有词都算功劳，失败了全部算错误。简单、好上手、出效果，但粒度太粗。
- **DPO/BPO**：更细一点，可以给每个词、每句话分别打分，但需要额外训练一个评分模型，带来不稳定性。
- **其他方法**：让更强的模型来读轨迹，找到关键转折点再做反思。

#### 宋博团队想解决的两个新问题

**问题一：语言的信息密度不均匀**

传统 RL 把所有词一视同仁，但语言中的词并非等价。

> 举例："然后……的……这个……"这些连接词几乎没有信息量；而"返回 null 而不是抛出异常"这样的表述信息量极高。

用词的数量来衡量"奖励信号应该传多远"，相当于把"然后"和"null"算成一样重要——这不合理。

**问题二：生成训练数据太贵**

On-policy RL 的意思是：每次训练前都要让模型自己先跑一遍任务，产生新数据，再用这些数据来训练。这就像每次考试前都要先做一套全新的模拟题，成本极高。学生/小团队根本负担不起。

#### 工作一：IM PPO（信息流形 PPO）

**做了什么**：改变"奖励信号如何往前传播"的规则。

传统做法：奖励信号按词数往前衰减，走 100 个词就衰减到原来的几分之一。

IM PPO 的做法：遇到信息量低的词，奖励信号**不衰减**，直接原封不动传过去；遇到信息量高的词，才把这个奖励信号"落地"，更新这里。

**怎么判断信息量高低**：用"熵"（Entropy）。熵是衡量不确定性的指标，一个词的熵越高，说明模型在这里越"纠结"，信息含量越大。这个直觉来自一篇研究，发现大约 20% 的词贡献了 80% 的语义信息，其余大多是低熵连接词。

**局限性**：熵在数学任务上表现好，但在其他任务上不一定可靠，这只是一个起点。

#### 工作二：Off-policy 混合训练

**做了什么**：不每次都生成新数据，而是把过去积累的"旧经验"也利用起来。

类比：学生不用每次都做新题，可以把以前做过的题目重新翻出来复习——但要筛选，太旧的经验可能不适用了。

**筛选的两个标准**：
1. **这段经验质量怎么样？** 成功的经验优先
2. **这段经验和当前模型差异有多大？** 差异太大的经验可能有误导，降低优先级

**工程意义**：训练端和采样（生成数据）端可以完全异步运行，中间用一个"数据缓冲池"协调，大幅降低训练成本。

### Level 2：对"局面"本身建立理解

#### 从 Level 1 到 Level 2 的升级

Level 1 还是在看"这条路走下去好不好"。Level 2 更进一步：**不管走哪条路，这个出发点（状态/局面）本身是好是坏？**

> 还是下棋的例子：Level 1 是"这一步棋走对了"；Level 2 是"当前这个棋局形势对我有利还是不利"——不依赖具体走哪步来判断。

这样的好处是：摆脱了"路径依赖"——不会因为你之前恰好走了某条路才得出这个结论，而是从局面本身出发做判断。

#### 现有方法的问题（trajectory-level reflection）

市面上大量工作让 AI 自我反思：完成一次任务 → 反思哪里做得不好 → 再重试。但这类方法有两个问题：

1. **路径依赖**：反思结论受到"我是怎么走到这一步的"影响，换条路可能得出不同结论
2. **冗余（redundancy）**：反思内容越来越多但信息量递减，用这些冗余反思去训练反而有害

#### 工作三：Active Visual Agent Reasoning（主动视觉推理）

**场景**：遥感图像识别——图像太大，AI 一次看不全；但如果切小了看，又会丢失上下文信息。

**做法**：让 AI 自己决定"截哪块图来看"，每次截图都要最大化"我能获得的新信息量"。

**关键概念：Belief（信念/认知状态）**

Belief 是 AI 对"答案在哪里"的概率分布。

> 开始时，AI 觉得目标可能在图像任何地方，belief 是均匀分布的（哪里都有可能）。截了左上角没找到，belief 就更新了：左上角可能性降低，其他区域可能性升高。再截一块，再更新。

这个 belief 驱动每次截图决策，形成循环：截图 → 更新 belief → 基于新 belief 决定下一次截哪里 → 再截图……

**优点**：不需要训练，完全基于推理迭代，成本随着迭代次数增加而增加，但效果也随之提升（一种"时间换性能"的缩放方式）。

#### 工作四：Stable Reflection（稳定状态反思）

**做了什么**：从同一个出发点（状态）让 AI 走多条不同的路，观察这些路的结果，然后更新对这个出发点的理解。

**Prior 和 Posterior（先验和后验）**：
- **Prior（先验）**：还没走任何路之前，我对这个局面的判断
- **Posterior（后验）**：看了多条路的结果之后，我对这个局面的新判断

这两者不同：看到经验后，AI 对"哪个动作好、哪个动作坏"有了更准确的认识，把这种认识变成文字表达出来，就是对当前状态的一个更稳定的理解。

**一个有趣的涌现现象：Explore vs Exploit 自发出现**

Explore（探索）= 尝试不同的路，看看有没有更好的  
Exploit（利用）= 坚持走已知最好的路

训练时只给最终奖励，没有告诉 AI 该探索还是利用。但实验中发现：
- 训练早期：AI 评估能力弱，搜索多了反而乱，所以自动收窄搜索范围
- 训练中期：专注提升自己的评估能力，整体性能甚至暂时下降
- 训练后期：评估能力强了，搜索范围自动放开，性能持续上升

这个平衡完全是**自发涌现**的，没有人为设计。

**Self-Distillation（自蒸馏）**：把后验理解"蒸馏"回先验里。

> 蒸馏 = 把老师的知识压缩进学生。这里是把"看过很多经验后的自己"教"还没看过经验的自己"，让模型一开始就带着更成熟的判断去做决策。

在国际象棋场景验证：模型能够把"下几步后的局面判断"提前融入当前决策，不再只看眼前一步。

### Level 3：知识层面的 Bootstrapping（开放问题）

宋博坦承这一层**目前没有具体完成的工作**，但提出了三个核心问题：

1. **怎么衡量"知识"？** 现在大多数工作连知识的定义都没有
2. **三个层次怎么互相关联？** Level 1、2、3 之间如何系统地串联起来？
3. **知识怎么提取和存储？** 继续预训练？微调？还是用检索生成的方式？

他提到，前面国际象棋实验里模型对"局面"的理解，已经有点像对知识的应用了——算是 Level 3 的一个雏形。

### 宋博的愿景：Academic Agent（学术智能体）

宋博最后分享了自己对未来 agent 的构想：一个能帮人类**管理知识、判断知识、获取知识**的 agent。

它需要具备：
- **Active 能力**：主动去找信息，而不是被动等待
- **Assessive 能力**：对知识有自己的判断，知道什么重要、什么不重要
- **Internal belief**：对世界有自己的认知模型
- 意识到：它的判断会影响用户获取知识的方式（normative influence）

## 第二部分：邓博 — Token 经济学与 Context Scaling

### 背景：KV Cache 是什么，为什么重要？

大模型每次处理输入时，会计算并存储一些中间结果（叫做 KV Cache，Key-Value 缓存）。如果同样的输入内容再次出现，就可以直接复用这些缓存，不用重新计算——**命中缓存的 token 比不命中的便宜很多**。

API 计费通常分为：
- **Prefill（输入处理）**：处理你发给模型的内容，比较贵
- **Decode（生成输出）**：模型生成回答，也贵
- **Cache 命中**：命中缓存的 token，价格只有前两者的几分之一到几十分之一

**关键发现**：同一个模型（DeepSeek V3.2），不同供应商的缓存命中率天差地别：
- **DeepSeek 自家**：命中率 87.1%（最便宜）
- **部分 Router 平台**（中间商）：命中率接近 0%（全额付费）

原因：自家最了解自己的模型，知道怎么构造输入能命中缓存；Router 平台完全不知道。

### Token 的价值因行业而异

不是所有 token 值一样的钱。核心区别在于输入/输出的比例：

- **代码生成、翻译**：输入固定（代码库/文章），输出是结论，缓存命中率高，整体便宜
- **药物研发、法律审阅**：每个案例都不同，需要大量个性化推理，输出长且无法复用，成本高

**结论**：同样百万 token，不同行业实际花费可以差几倍。

### Thinking Mode 的效率问题

邓博做了个实验：同一道从未见过的 IMO 数学题，同一个 20B 参数的模型：
- **Thinking 设为 High**：思考了 270 秒（4.5 分钟），答对了
- **Thinking 设为 Low**：几秒钟，也答对了

**结论**：大量的"思考"是多余的。模型足够强时，不需要那么多思考就能答对。

未来 RL 训练的新方向：不是训练模型答对更多题，而是**训练模型用更少的思考步骤答对题**——从"数值对齐"转向"思考过程对齐"。

北大有个工作叫 LAMPO，用自然语言和数值同时作为奖励信号来做这件事，效果很好。

### 五层蛋糕：整个 AI 系统的成本层级

| 层级 | 内容 | 优化方向 |
|------|------|---------|
| 芯片（Hardware） | 物理算力 | 降低功耗，减少 IO |
| 推理引擎（Engine） | TensorRT 等，含 KV Cache 优化 | 提高并发，减少重计算 |
| 模型（Model） | 通过 RL 让模型思考更精简 | 缩短 thinking 链 |
| Agent 框架（Agent） | Harness，协调模型完成复杂任务 | 减少冗余调用 |
| Context | 用户需求产生的上下文 | 最大化缓存命中 |

类比操作系统：每个线程都有上下文（Context），CPU 要花时间处理这些上下文切换。AI 系统也一样，管理 context 的成本不可忽视。

重要提醒：**不是所有任务都值得走 agent**。用 Claude Code 说声 "hello" 花了 23 块——关个灯值不值得这个钱？

### Token Efficiency 框架

核心公式：**Token Efficiency = 用户价值 / Token 成本**

Token 不直接产生价值，要经过三次转换才能到达用户：

```
Token → 决策（模型做的事）→ 任务进展（Agent 执行）→ 用户价值（产品）
```

每一步都有损耗。而且：
- Token 成本不是线性的，用了缓存就便宜很多
- 前面的 token 很重要，后面的 token 边际价值递减

**最优 token 数量的判断标准**：再多一个 token 带来的价值 = 这个 token 的成本。超过这个点继续加 token，就是在浪费钱。

邓博估算，这个峰值大约在 20K~30K token 左右——而目前没有任何 agent 框架能稳定消耗 3 万 token 还持续产生价值，说明当前 agent 系统**远未达到效率最优点**，还有很大空间。

### 工作一：Context Reorder（MLSYS 论文）

**问题**：RAG（检索增强生成）场景中，系统从知识库检索出几篇文档，每次检索顺序不同。模型把文档按不同顺序放进输入，就无法命中缓存——哪怕文档内容完全一样。

**解法**：把检索到的文档按**固定顺序**排列（用"最长公共子序列"算法），不管检索顺序怎么变，都整理成同一个固定序列放给模型。

> 类比：图书馆的书总是按编号排列，不管读者以什么顺序取书，还回去的时候都按统一顺序归架，下次来就能直接找到。

然后在输入末尾补充一小段文字，告诉模型"原始的相关性顺序是什么"，保证效果不受影响。

**效果**：质量损失只有 0.1%~1.9%，速度大幅提升。

| 框架/场景 | 提速 |
|-----------|------|
| Multi-agent（COO 框架） | 2 倍 |
| Memory0（agent 记忆框架） | 1.83 倍 |
| Claude Code（官方） | prefill 延迟降低 63%，整体约 1.3~1.4 倍 |
| 多轮 RAG 对话 | 3 倍以上 |
| MacBook Air M3（本地） | 2.41 倍 |
| Jetson Orin（嵌入式） | 1.5 倍 |

**为什么行得通？**：LLM 和人脑一样，对顺序有很强的鲁棒性。我们看到"WAHT"也知道是"WHAT"；句子主谓宾颠倒也能理解。打乱文档顺序，模型照样能 recall 原始的语义关系，只要告诉它原始顺序是什么就行。

**这意味着**：agent 的 memory（记忆）和 skills（技能）模块的存储顺序，完全不需要跟使用顺序一致——这大大简化了 context 管理设计。

已开源，即插即用，**不需要修改任何推理引擎或 agent 框架的代码**。

### 工作二：Visual Token Pruning（ACL 论文）

**问题**：处理高分辨率图像或视频时，视觉 token 数量爆炸式增长（视频尤其严重），而 attention 计算量是 token 数量的平方——100 个 token 变 200 个，计算量变 4 倍。

**解法**：在给定计算预算下，动态决定哪些视觉 token 可以丢掉。

> 就像读一篇文章，你会跳过那些"废话"段落，专注于信息密集的部分——AI 处理图像时也可以这样。

**结论**：原始模型认为所有 token 都需要，但实验证明删掉大量 token 后效果几乎不变。很多 token 是"伪需求"——它们只是凑数的，没有真正的信息量。

### 实验：100 英镑能换来多少价值？

- **任务**：70 个来自 Claude Task 测评集的任务
- **框架**：Claude managed agents（含 planner 和 SDP computer）
- **模型组合**：随机搭配 3~4 家不同模型
- **预算限制**：只有 100 英镑，在预算内尽量完成更多任务

结果：GPT-5.4 + Minimax 组合得到最高准确率（ACC≈0.8），即 100 英镑换来了约 125 单位的经济效用。

**算出来的结论**：每个 token 实际产生的价值，**远低于 API 供应商的计价**——他们赚了很多"躺赚"的差价。

邓博的评价："广大龙虾养殖人员，你们花的钱实在是太多了。"（指用 API 的开发者）

## Q&A 精华

**Q（观众）：怎么让 AI 在 RL 训练时思考得更高效，减少废话？**

A（宋博）：核心是对反思内容做"后处理/清洗"，而不是任由反思越来越多。

具体层次：
1. 把相同"局面"出发的多条轨迹聚合 → 得到对这个局面的稳定认识
2. 再把多个局面的稳定认识 → 浓缩成任务级别的知识（比如"这类任务经常犯什么错误"）
3. 最终变成可复用的精简文本，而不是越堆越多的冗余反思

**Q（观众）：IM PPO 里高熵/低熵的权重策略没理解**

A（宋博）：传统 PPO 是按词数往前传奖励信号，走 100 个词衰减到几乎为零。我改成按信息密度传：经过低熵词（"然后"、"的"）时信号不衰减，原封不动往前传；遇到高熵词（关键决策词）才把这个信号"落地"更新这里。低熵词不参与更新，相当于被跳过了。

**Q（观众）：Token 效率优化到什么程度算合理？**

A（邓博）：参考指标是 KV Cache 命中率。命中率达到 **80%** 以上，就基本对标了 DeepSeek 自家的水平（87.1%），可以认为优化到位了。判断方法：把 decode 阶段的 KV Cache 拿出来，和历史对话的 KV Cache 算相似度，匹配到 80%~90% 就 OK。

**Q（观众）：Context Reorder 已经有其他公司在做延伸了吗？**

A（邓博）：很多人在做。理想的优势是领域很垂（车和家），context 重复度天然高，命中率本来就不会差。他们现在从 document 级别的 reorder 做到了 memory 级别（Memory0），下一步做到 sentence 级别，最终目标是 token 级别（就像"孙悟空和猪八戒"和"猪八戒和孙悟空"对理解没影响）。

云端和端侧的策略不同：云端靠 reorder 提高命中率；端侧（本地设备）reorder 意义不大，但多轮对话的 KV Cache reuse 很有价值。

最后邓博补充了一个"道德问题"：如果你已经命中了 KV Cache，但还是按原价向用户收费——这也是一种赚钱方式。（言下之意，目前很多 API 供应商都在这么干。）
