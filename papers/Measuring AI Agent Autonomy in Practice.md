# Measuring AI Agent Autonomy in Practice

**来源**: https://www.anthropic.com/research/measuring-agent-autonomy  
**发布时间**: 2026 年 2 月 18 日  
**数据来源**: Claude Code 使用数据 + 公开 API 工具调用（约 100 万条）  
**隐私工具**: CLIO（隐私保护分析系统）

---

## 一句话概括

首次基于真实部署数据系统性测量 AI agent 的自主度：Claude Code 最长运行时间三个月内翻倍，有经验的用户同时提高自动批准率和打断率——监督策略从"逐步审批"转向"异常干预"。

---

## 四项核心发现

| 发现 | 数据 |
|------|------|
| Claude Code 自主运行时长翻倍 | 99.9th 百分位从 <25 分钟增至 >45 分钟（3 个月内） |
| 有经验用户的全自动审批率翻倍 | 新用户 ~20%，750 sessions 后 >40% |
| 有经验用户打断率也上升 | 新用户 5%，有经验用户 ~9% |
| Claude 自主暂停频率高于人类打断 | 复杂任务中 Claude 提问频率 >2× 人类打断频率 |

---

## 数据来源与方法

### 两个互补数据源

| 数据源 | 优势 | 局限 |
|--------|------|------|
| **公开 API**（~100 万 tool calls） | 覆盖数千个客户的多样化部署 | 只能分析单条 tool call，无法重建完整会话 |
| **Claude Code**（完整会话） | 可追踪完整工作流：自主时长、打断原因、用户策略演变 | 只有编程一个场景 |

**Agent 定义**：配备工具（运行代码、调用 API、向其他 agent 发消息）的 AI 系统。

---

## 发现一：自主运行时长在增长

**中位数 turn 时长**：约 45 秒，过去几个月变化不大（40–55 秒）——这符合预期，新用户持续涌入会拉低平均值。

**关键信号在尾部**：
- 99.9th 百分位 turn 时长：2025 年 10 月 < 25 分钟 → 2026 年 1 月 > 45 分钟
- 增长**平滑穿越模型版本更新节点**，说明不只是模型能力提升的结果，更反映：用户建立信任、任务越来越复杂、产品本身持续改进

**与 METR 能力评测的对比**：
- METR 评测：Claude Opus 4.5 能完成人类需要 5 小时的任务（50% 成功率）
- Claude Code 实际：99.9th 百分位约 42 分钟，中位数更短
- 两者不矛盾：METR 测的是理想条件下的能力上限，实际数据包含了 Claude 主动暂停和用户打断

→ **实际授予的自主度显著低于模型能够处理的能力上限**（deployment overhang）

**Anthropic 内部数据（Aug–Dec 2025）**：
- 最复杂任务的成功率翻倍
- 平均人类干预次数：从 5.4 次降至 3.3 次

---

## 发现二：有经验用户的监督策略转变

**表面矛盾**：有经验用户自动批准率↑（更少逐步审批）但打断率也↑（更多主动干预）

**解释**：这不是矛盾，而是监督策略的根本转变：

| 用户类型 | 策略 |
|---------|------|
| **新用户** | 逐步审批每个 action，很少需要打断 |
| **有经验用户** | 让 Claude 自主运行，异常时才介入 |

从"预防性审批"→"异常检测干预"，后者需要更强的态势感知能力和对何时介入的判断力。

**API 数据的印证**：
- 低复杂度任务（如修改一行代码）：87% 的 tool call 有人类参与
- 高复杂度任务（如自主找零日漏洞、写编译器）：仅 67% 有人类参与

---

## 发现三：Claude 是监督的主动参与者

Claude 会自主暂停请求澄清，不只是等人类干预：

- 复杂任务中 Claude 提问频率 > 无 AI 打断频率的 2 倍
- 任务越复杂，Claude 越主动寻求确认

### Claude 暂停的常见原因

| 原因 | 占比 |
|------|------|
| 提出多个方案供用户选择 | 35% |
| 收集诊断信息或测试结果 | 21% |
| 澄清模糊或不完整的请求 | 13% |
| 请求缺失的凭证/权限 | 12% |
| 执行前获取确认 | 11% |

### 人类打断 Claude 的常见原因

| 原因 | 占比 |
|------|------|
| 提供缺失的技术上下文或纠错 | 32% |
| Claude 太慢、卡住或过度 | 17% |
| 已获得足够帮助可以独立继续 | 7% |
| 想自己做下一步（手动测试、部署等） | 7% |
| 中途修改需求 | 5% |

---

## 发现四：Agent 在高风险领域有使用，但规模尚小

### 当前 API 工具调用的整体风险画像

| 指标 | 数据 |
|------|------|
| 有某种安全防护措施的 tool call | 80% |
| 有人类在环的 tool call | 73%（可能是上限，存在过估） |
| 不可逆操作（如发客户邮件） | **仅 0.8%** |

### 领域分布

- **软件工程**：约 50% 的 tool call
- 商业智能、客服、销售、金融、电商：各占几个百分点
- 医疗、网络安全：出现但规模很小

### 高风险/高自主度集群示例

**高风险**：实现 API 密钥窃取后门、访问患者医疗记录、部署生产 bug 修复
**高自主度**：红队渗透测试、自动执行加密货币交易、自动发送会议提醒、监控邮件

注：研究者认为许多"高风险"操作实际上是安全评测（红队测试），而非真实生产操作。

## 政策建议

### 对模型和产品开发者

1. **投资部署后监控**：预评测只能测能力，无法捕捉真实使用模式。需要构建隐私保护的实时监控基础设施。
2. **训练模型识别自身不确定性**：让模型主动暂停请求澄清，是外部防护（权限系统、人类审批）的重要补充。
3. **为用户监督而设计产品**：提供可信的 agent 行为可见性 + 简单的干预机制（参考 Claude Code 的 real-time steering 和 OpenTelemetry 集成）。

### 对监管者

**不要强制规定特定的交互模式**（如要求批准每个 action）——这会产生摩擦而未必带来安全收益。焦点应放在"人类是否处于能有效监控和干预的位置"，而非要求特定的参与形式。

---

## 核心结论

> Agent 在实际部署中表现出的自主度，是**模型、用户、产品三者共同构建**的结果——不是模型能力单独决定的。

- Claude 通过主动提问限制自身自主度
- 用户随经验积累调整监督策略
- 产品设计（默认需审批 → 可配置自动审批）塑造了用户行为

这意味着仅靠预评测无法全面理解 agent 行为，**部署后的实时测量基础设施**是 AI 安全的必要组成部分。

## 个人理解

这篇对我作为 Claude Code 用户有直接的实践含义：

**有经验用户的"自动批准+更多打断"模式**描述了一种成熟的工作方式——不是放任 Claude 随意做，而是建立了足够的信任后放手让它工作，同时保持对异常信号的敏感度。这是一种需要主动培养的技能：知道什么情况下要打断、什么情况下可以继续等待。

**deployment overhang 的概念**值得记住：模型实际能处理的自主度远超用户实际授予的自主度。这不一定是坏事——谨慎建立信任是合理的，但也意味着有经验后可以更放心地给更长的任务。

**Claude 主动暂停的最常见原因（35% 是提供方案选择）**说明 Claude 在重要决策点倾向于停下来问，而不是自己做选择。这与"plan mode"的使用动机是一致的——复杂任务前先对齐方向，减少后续需要打断的次数。
