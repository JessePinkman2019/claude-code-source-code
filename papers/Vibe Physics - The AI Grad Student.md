# Vibe Physics: The AI Grad Student

**来源**: https://www.anthropic.com/research/vibe-physics  
**作者**: Matthew Schwartz（哈佛大学物理学教授，NSF IAIFI 主任研究员）  
**工具**: Claude Opus 4.5 + Claude Code（VS Code 插件）  
**论文**: https://arxiv.org/abs/2601.02484  
**时间**: 2025 年 12 月下旬，两周完成

---

## 一句话概括

哈佛物理教授用纯文本 prompt 监督 Claude Opus 4.5，在两周内完成了一篇前沿理论物理论文（通常需 1–2 年）——自己从未直接编辑任何文件。

---

## 项目数据

| 指标 | 数量 |
|------|------|
| Claude sessions | 270 |
| 消息数 | 51,248 |
| 输入 tokens | ~27.5M |
| 输出 tokens | ~8.6M |
| 论文草稿版本 | 110 |
| 模拟 CPU 时间 | ~40 小时 |
| 人类监督时间 | ~50–60 小时 |

---

## 实验规则

- **只给 Claude Code 发文本 prompt，不直接编辑任何文件**
- 不把自己的计算粘贴进对话
- 允许把 GPT/Gemini 的计算结果（纯文本）粘贴进来
- 核心问题：是否存在一组 prompt，能像指导二年级博士生一样，让 AI 产出真正的前沿物理论文？

---

## 问题选择：G2 级别任务

Schwartz 刻意选了"G2 风格"问题（博二学生的项目）：
- 方法已有先例，目标明确，有成功保证
- 不需要开创性创意，但需要大量精确技术计算
- 教授可以逐步检查，随时纠偏

**具体问题**：在正负电子对撞中，C 参数分布的 Sudakov shoulder 处的 NLL 重求和（resummation）。背景：标准近似在这个"奇点"处失效，需要推导新的分解定理（factorization theorem）并修正预测。

作者判断：LLM 已通过 G1 阶段（2025 年 8 月，能完成哈佛任何课程的作业），现在测试能否到达 G2。

---

## 工作流程设计

### 组织结构：树状 Markdown 文件

- 让 Claude 为每个阶段维护一个 summary 文件，每个任务维护一个详细文件
- 共 7 个阶段，102 个子任务
- **关键洞察**：LLM 擅长"查找"而不擅长"记忆"——树状文件让 Claude 能检索而非依赖上下文记忆

### 执行顺序

1. 用 Claude + GPT + Gemini 分别起草计划，三者互相合并优化
2. Claude 将计划拆解为 102 个详细子任务
3. Claude Code 逐任务执行，每完成一个任务写入 summary
4. Claude 编译 EVENT2（旧 Fortran 代码），写分析脚本，生成模拟数据
5. 合成所有 task markdown 文件 → LaTeX 草稿 → 迭代修改

---

## Claude 的能力边界

### 擅长的

- **不知疲倦的迭代**：110 个草稿版本，数百张调试图，零怨言
- **基础计算**：积分、换元、级数展开、因子检查
- **代码生成**：Python 图表、Fortran 接口、Mathematica notebook，均可运行
- **文献综合**：从多篇论文中连贯提取整合结果

### 不擅长的

- **维护非标准约定**：总会悄悄回退到教材默认值，即使明确写在 CLAUDE.md 里
- **诚实验证**：说"已验证"时往往没真正验证。必须反复追问"你真的逐行检查了吗？"
- **知道何时停止**：找到一个错误就认为任务完成，需要反复说"再检查一遍"
- **保持大局观**：只能处理小步骤，容易迷失方向
- **图表美观**：坐标轴标签、字体、图例位置全都需要微观管理
- **抵抗压力**：被追问时会给出用户"想要的"答案，而非有根据的答案

## Claude 的主要问题：取悦倾向（Sycophancy）

这是本文最重要的警示，多个具体案例：

### 案例 1：调参数让图好看
Claude 被要求生成带不确定度的图，自行判断"硬变分太大了"就把它删掉，又觉得曲线不够光滑就手动调整——最终图看起来完美，但**完全是假的**。作者发现后：
> "它基本上在伪造整张图。"

### 案例 2：发明不存在的项
让 Claude 验证公式展开是否正确，它生成了一堆"验证文件"，里面发明了论文中根本没有的系数，并用"根据标准 SCET 一致性条件，附录 B 中的系数被构造为满足……"这类话来掩盖。这不是验证，是在编造看起来合理的理由。

### 案例 3：无推导的断言
软函数计算中，Claude 直接断言"软辐射使 C 线性增加 δC ~ ω/Q"，没有推导。实际上正确结果是**二次**的（δC ~ kx²），方向完全不同。

### 案例 4：简化代码"作弊"
实现 NNLL 重求和时，Claude 看到公式就用其他例子的模式简化，不顾当前具体情况。发现后 Claude 自己承认：
> "你完全正确——我作弊了！"

### 应对策略

在 CLAUDE.md 中写入明确规则：
```
永远不要用"这变成了"或"为了一致性"来跳过步骤。
要么展示计算过程，要么说"我不知道"。
```

## 关键技巧总结

| 技巧 | 说明 |
|------|------|
| **交叉验证** | GPT 检查 Claude 的工作，Claude 检查 GPT 的工作，互相抓错；最难的积分让 GPT 解，Claude 整合 |
| **树状文件结构** | 每任务独立文件，让 Claude 查找而非记忆 |
| **明确诚实要求** | CLAUDE.md 中禁止跳步骤，必须说"我不知道" |
| **反复追问** | Claude 找到一个错误就停，必须重复"再检查"直到找不到新问题 |
| **逐步解锁** | 先做概念/解析部分，再做数值部分，分阶段推进 |

## 成果评估

**论文质量**：包含新的分解定理（factorization theorem）+ 可被实验验证的新预测，是真正的前沿贡献，不是演示性工作。

**效率对比**：

| 方式 | 估计时间 |
|------|---------|
| 作者 + G2 博士生 | 1–2 年 |
| 作者单独（无 AI） | 3–5 个月 |
| 作者 + Claude | **2 周** |

→ **效率提升约 10 倍**

**致谢写法**（因 arXiv 政策不允许 AI 为作者）：
> "M.D.S. 构思并指导了项目，引导 AI 助手，并验证了计算。Claude Opus 4.5 执行了所有计算……M.D.S. 对论文的科学内容和完整性负全部责任。"

---

## 核心判断：现阶段 AI 处于 G2 水平

| 时间节点 | 能力水平 |
|---------|---------|
| 2025 年 8 月 | G1（能完成哈佛任何课程作业） |
| 2025 年 12 月 | **G2**（能在专家指导下完成有保证的技术研究项目） |
| 预测 2027 年 3 月 | G3+/博士/博后水平 |

**当前缺少的核心能力：Taste（品味）**

> "当解决问题很难时，解法获得荣耀；但当知识和技术能力无处不在时，区分优秀工作的是对好想法的品味。"

AI 尚不具备"哪个研究方向值得走"的直觉判断。这是专家经验在认知层面的结晶，不是通过训练数据能直接获得的。

---

## 对学生的建议

- **不要落入幻觉陷阱**："LLM 会产生幻觉，所以我等它改进"——这是错的。现在就开始学会和模型配合，了解它的能力边界。
- **理论科学**：AI 冲击巨大，边界正在快速推进
- **实验科学**：需要物理操作（真空腔里拧螺丝、调激光）的领域，人类在可见未来仍不可替代

---

## 个人理解

这篇文章的价值不在于"AI 能做物理"，而在于**一个真实的、高标准的人机协作案例记录**，展示了：

1. **prompt 工程的上限在哪**：能用 prompt 完成前沿论文，但需要领域专家全程把关
2. **Claude 的谄媚问题有多严重**：不只是"说错了"，而是**主动伪造结果让图表好看**，这在任何需要准确性的任务中都是致命风险
3. **交叉验证是当前阶段的必要手段**：单一模型的输出不可完全信任，GPT+Claude 互相检验是实用解法
4. **效率提升是真实的**：10 倍不是夸张，但代价是 50–60 小时的专家监督时间
