# `/simplify` 与 `/batch`：内置多 Agent 工作流命令

两个命令均在 Claude Code v2.1.63 发布，解决了此前需要精心编写多步 prompt 才能完成的两类工作流。

---

## `/simplify`：实现后的三 Agent 代码审查

### 它做什么

实现功能后运行 `/simplify`，它自动：

1. 运行 `git diff`（或 `git diff HEAD`）找出变更文件，无 git 变更则回退到最近修改的文件
2. **并行启动三个专项审查 Agent**，各自接收完整 diff：

| Agent | 关注点 | 具体检查 |
|-------|--------|---------|
| **代码复用** | 重复逻辑、冗余模式 | 现有工具函数可替代的新代码、跨文件重复函数、手写的字符串/路径处理（已有 helper）、应使用现有抽象的内联逻辑 |
| **代码质量** | 可读性、结构、规范 | 冗余状态、参数膨胀、略有差异的复制粘贴、泄漏抽象、使用原始字符串而有类型常量可用的"字符串类型化"代码 |
| **效率** | 性能和资源使用 | 不必要的工作、错失的并发机会、热路径臃肿（紧循环里的昂贵逻辑）、TOCTOU 反模式、内存泄漏、读取整个文件但只需要部分的过宽操作 |

3. 三个 Agent 同时运行，完成后聚合结果
4. 直接修复每个有效问题，静默跳过误报
5. 输出简短摘要，或确认代码已经干净

### 使用时机

**最佳时机：实现后、开 PR 前**

- 完成功能后 → PR 前清理
- 修完 bug 后 → 确保修复没引入捷径
- 原型完成后 → 收紧想保留的实验性代码
- PR 前 → 提前捕获审查者会标记的问题

可选传入文本聚焦审查范围：
```
/simplify focus on the auth middleware changes
/simplify check the database query efficiency
```

### 它不是什么

- 不是 linter，不是 formatter——它在更高层次操作：架构决策、代码结构、算法效率
- 不是通用重构工具——只聚焦最近变更的文件，不扫描整个代码库
- 与 ESLint/Biome 互补，不替代

## `/batch`：并行代码库迁移

### 三阶段编排流程

描述要做什么，`/batch` 触发一个深度编排 Agent，处理研究、分解、执行和 PR 创建。

#### 阶段一：研究与规划

编排器进入 Plan 模式，启动 Explore Agent 深度研究指令涉及的内容——找出所有需要变更的文件、模式和调用点。然后将工作分解为 **5~30 个自包含单元**（根据代码库大小和变更复杂度）。

每个单元的要求：
- 可在独立 worktree 中独立实现
- 可单独合并，不依赖其他单元先落地

编排器还会确定端到端验证方案（浏览器自动化、CLI 测试或现有测试套件）。如果无法确定验证路径，在继续前会询问你。

#### 阶段二：并行启动 Worker

你批准计划后，编排器**在单条消息块里启动所有 Worker Agent**，实现真正并行。每个 Agent：
- 获得 `isolation: "worktree"`，在独立 git worktree 中工作
- Prompt 完全自包含：整体目标 + 具体任务 + 代码库规范 + 验证方案
- 实现后自动运行 `/simplify`
- 执行测试套件
- commit、push，用 `gh pr create` 开 PR

#### 阶段三：追踪进度

编排器渲染状态表，随 Agent 完成实时更新，从每个 Agent 输出中提取 PR URL。最终给出类似"22/24 个单元已作为 PR 落地"的汇总。

### Worktree 隔离是关键

每个 Agent 有自己的分支和工作副本，互不干扰变更，不会产生合并冲突。每个单元可独立测试和审查。

### 计划审批步骤

执行任何代码变更前，`/batch` 展示完整计划：每个工作单元、会触碰的文件、变更内容。**你批准计划后 Agent 才开始执行**。

计划看起来有问题（漏了某些文件、分组不合理），可以在执行前调整。

### 前置条件

必须在 git 仓库中。每个 Agent 在独立 git worktree 运行，每个单元完成后开 PR。不在 git repo 里，`/batch` 不会运行。

### 适合的工作

| 适合 | 不适合 |
|------|--------|
| 框架迁移（class 组件 → 函数组件）| 文件 A 依赖文件 B 新输出的紧耦合变更 |
| API 契约变更（更新所有调用方）| 还不知道终态的探索性重构 |
| 规范强制执行（命名规范、错误处理、模式标准化）| 不需要分解的单文件变更 |
| 依赖替换（一个库替换成另一个）| |
| 测试生成（为未测试模块添加覆盖）| |

**如果变更单元之间需要协调**（Agent A 创建共享工具，Agent B 需要 import 它），`/batch` 不是正确工具——用标准 Claude Code session 或 agent teams。

## 两个命令的对比

| 维度 | `/simplify` | `/batch` |
|------|------------|---------|
| **规模** | 最近变更的文件 | 整个代码库 |
| **时机** | 实现之后 | 实现之前 |
| **工作类型** | 审查并改进现有代码 | 在多文件中应用变更 |
| **Agent 数量** | 3 个并行审查者 | 5~30 个并行实现者 |
| **输出** | 直接应用修复到当前分支 | 多个 PR（每个单元一个）|
| **Git 要求** | 否 | 是（使用 worktree）|
| **审批步骤** | 审查 diff | 批准计划，然后审查 PR |
| **最适合** | 精炼 | 迁移 |

**心智模型**：`/simplify` 是你的代码审查者，`/batch` 是你的迁移团队。

### 内置集成

`/batch` 的每个 Worker 在 commit 前自动运行 `/simplify`。`/batch` 生成的每个 PR 都已经经过了三 Agent 审查，无需手动串联。

## 实用技巧

**每次 PR 前都跑 `/simplify`**：养成习惯。三 Agent 审查能捕获你在数小时专注实现后容易忽视的问题。实践中 `/simplify` 每个功能分支稳定能找到 3~5 个问题，效率 Agent 尤其擅长发现不必要的迭代和错失的并发机会。

**`/batch` 描述要具体**："更新代码库"太模糊。"将所有 moment.js 导入替换为 dayjs，更新 API 调用以匹配 dayjs 语法"给规划器足够的上下文准确分解。

**仔细检查 `/batch` 计划**：分解步骤是大多数问题浮现的地方。如果某个单元跨越太多文件或混合了独立变更，在批准前要求重新拆分。

**配合测试套件使用 `/batch`**：每个 Worker Agent 在变更后跑测试。确保你的测试套件真的能捕获回归，弱测试意味着 Worker 会"通过"损坏的代码。

## FAQ

**`/simplify` 会自动修改文件吗？**
是的，直接应用修复到工作副本。用 `git diff` 审查变更后再 commit。如果某个修复有误，`git checkout` 对应文件回退。

**`/batch` 的某个 Worker 失败了怎么办？**
失败的 Worker 不影响其他 Worker，各自在独立 git worktree 和分支上运行。状态表里能看到失败，可以重试该单元或手动处理。已成功完成的 PR 不受影响。

**`/batch` 支持 monorepo 吗？**
支持。在描述中明确指定要处理的 package 或目录，让规划器正确分解工作。

**`/simplify` 和 linter 有什么区别？**
Linter 捕获语法和风格问题。`/simplify` 捕获架构问题——重复逻辑、错失抽象、性能低效、人工审查者会标记的模式。两者互补。
