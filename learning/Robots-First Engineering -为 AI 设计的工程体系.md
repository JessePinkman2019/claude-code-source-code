# Robots-First Engineering：为 AI 设计的工程体系

## 核心认知转变

软件工程中所有东西都是为人类设计的：文件系统、终端接口、JSON、Git 工作流、代码审查流程。

**当我们为机器重新设计时，会发生什么？**

---

## 两件事的分离

**软件开发**已经自动化了。写代码、跑测试、提交变更——Agent 做这些，约 $10/小时，24/7 不间断，不会疲惫。

**软件工程**是设计让自动化开发保持可靠的系统：构建护栏、管理失败域、创建验证循环、设计反压机制。

工作重心已经转移。你不再亲手把货物搬上船，集装箱时代已经来了。**你的工作是驾驶机车，让它保持在轨道上。**

---

## 螺丝刀 vs 电锤：别跳过基础

**螺丝刀阶段**（手动基础）：
- 手动塑造规格
- 仔细管理 context
- 引导搜索和链接
- 在放手之前学会怎么让系统保持在轨道上

**电锤阶段**（全功率自动化）：
- Ralph 循环无人值守运行
- 大规模编排
- 多 Agent 并行
- 隔夜功能构建

**最常见的错误：在掌握螺丝刀之前就拿起电锤。**

不理解基础就直奔全自动化，会得到糟糕的结果。螺丝刀阶段教你电锤需要什么才能成功。先手动，理解为什么有效，然后升级到电动工具。

---

## Context 窗口是数组

这是 robots-first 思维的基础原则：

**context 窗口是一个数组。数组用得越少，窗口需要滑动的越少，结果越好。**

Context 越多意味着：
- 滑动越多
- 更早触发压缩
- 压缩是有损的
- 损失会丢掉关键锚点
- 漂移随之而来

这就是 Ralph 循环有效的原因——它们被设计为确定性地管理这个数组：保持聚焦、避免压缩、保留"锚点"。

**锚点**是你的规格说明——防止 Agent 发明的稳定参照点。每次循环迭代都引用它。没有锚点，Agent 就会漂移进入幻觉。

---

## 规格说明是查找表

Robots-first 的规格不只是文档，**它们是为 Agent 搜索优化的查找表**。

一个良好设计的规格包含：
- "也称为"（also known as）章节——提高搜索命中率
- 当 Agent 搜索"login"时，它找到 auth 规格
- 更多描述词 = 更多命中 = 更少发明

规格成为所有功能的参照框架。Agent 查找现有模式，而不是发明新的。

---

## 强链接胜过花哨格式

很多工程师过度设计实现计划：复杂 JSON schema、嵌套任务层级、精美格式。

Robots-first 的方法：**强链接胜过花哨格式。**

有效的做法：
- 带有显式引用的项目符号
- 引用精确的规格章节
- 指向具体文件位置
- 引用具体的代码块（Read 工具按块工作）

Agent 现在可以精确搜索，找到正确的代码块，遵循现有模式，不会发明东西。

---

## 为机器重新设计栈

如果你控制整个技术栈，可以为机器优化。以下是被质疑的人类优先假设：

| 人类设计 | 问题 | 机器优化方向 |
|---------|------|------------|
| Unix 用户空间和 TTY 约定 | Agent 需要终端接口吗？ | 移除终端仿真层 |
| JSON 数据格式 | JSON 对 tokenization 不是最优的，每个引号、括号、逗号都消耗 token | 二进制格式或 Agent 优化文本格式 |
| 敏捷/Scrum 流程 | Agent 需要 sprint 计划吗？ | 用验证循环替代人工评审 |
| 代码审查工作流 | 能用验证替代审查吗？ | 自动验证门控 |
| 垃圾回收 | 当"进程"是无状态 API 调用时，内存管理意味着什么？ | 重新设计假设 |

**Erlang/OTP 原则在这里变得相关**：Actor 模型、监督树、let-it-crash 哲学。这些模式非常适合 Agent 编排。

控制技术栈的工程师可以优化 tokenization、大幅降低成本、构建更快的响应式系统、比竞争对手运营成本更低。

## Loom：为 Agent 设计的环境

Loom 代表下一个演化：一个围绕 Agent 而非人类构建的自我演进软件环境。

设计原则，对每个系统组件问三个问题：
1. 这是为人类设计的吗？
2. 如果是，能删掉吗？
3. 如果删掉，损失了什么价值，如何替代？

Loom 架构包含：
- 代码托管（类 GitHub，但 Agent 优化）
- 使用 JJ 的源代码控制（为机器工作流设计）
- 类 Codespaces 的远程环境
- 多 LLM"合金化"（混合提供商）
- Actor/pub-sub 消息模式
- Erlang/OTP 风格的链式调用

**核心洞察：循环组合成链。循环套循环套循环。每个小而聚焦，合在一起形成响应式系统。**

---

## 终点：Weavers（织者）

Robots-first 工程的终极形态：**Weavers**——完全自主的 Agent：

- 在功能标志后面发布代码
- 无需代码审查直接部署
- 观察产品分析数据
- 自行决定变更是否修复了错误
- 自行决定是否进一步优化
- 自动迭代

**没有人在回路中。完全自主。**

这听起来很可怕——这是正确的初始反应。但考虑：

你的每一个顾虑都是一个工程问题：
- "如果它搞坏了生产怎么办？"→ 工程化回滚机制
- "如果它做了错误决策怎么办？"→ 工程化验证循环
- "如果它偏离轨道怎么办？"→ 工程化反压

**软件工程师的新工作变成：工程化掉那些顾虑。**

---

## 反压：让机车保持在轨道上

自主循环会创造很多失败域。你的工程工作变成构建护栏：

- 给规格添加约束
- 收紧计划链接
- 改善搜索指引
- 增加门控（测试、linting、格式化、安全检查）

当结果不好时，补救是另一个循环：
- 重构循环
- 规范循环
- 安全循环
- i18n 循环

反压让生成函数保持在轨道上。机车保持在轨道上，因为你工程化了轨道。

## 四个层级

| 层级 | 名称 | 特征 |
|------|------|------|
| **Level 1** | 手动基础 | 对话生成规格、手动编辑收紧约束、最小化 context、有人值守操作 |
| **Level 2** | 无人值守循环 | `while true` 风格运行、单目标迭代、自动化测试+commit/push、检查点状态 |
| **Level 3** | 多循环编排 | 多个循环同时运行、响应式链式调用、Actor/pub-sub 模式 |
| **Level 4** | 自主产品系统 | Weavers 发布功能、分析驱动决策、无需代码审查、完全产品自主 |

每个层级建立在前一个之上。不能跳过 Level 1-3 直接到 Level 4。

## 思维方式的转变

| 旧思维 | 新思维 |
|--------|--------|
| 我怎么写这段代码？ | 我怎么设计可靠地写出这段代码的系统？ |
| 我怎么审查这个 PR？ | 我怎么构建让审查变得不必要的验证？ |
| 我怎么调试这个问题？ | 我怎么工程化防止这类问题的反馈循环？ |

你不再是写代码的人，而是设计自主系统的人。**你是机车工程师，不是搬运工。**

---

## 从今天开始的五个问题

1. **我编码了哪些人类假设？** 审视你的工作流，哪些步骤存在是因为人类需要它们？

2. **什么可以变成查找表？** 你的规格、模式、规范——能为 Agent 搜索结构化吗？

3. **哪里的链接比较弱？** 你的计划可能引用了模糊的东西，能加上具体文件路径、行号、代码块吗？

4. **我的反压策略是什么？** 当 Agent 出错时怎么纠正？在需要之前先构建这些机制。

5. **完全自主需要什么？** 想象把自己从循环中移除，什么会崩溃？那些就是你的工程优先级。

---

## 经济现实

能运行可靠自主循环的团队和不能的团队之间的差距正在拉大。经济数据很直白：

- 自动化开发：约 $10/小时
- 人工开发者：$100+/小时

掌握 robots-first 工程的团队将以根本不同的速度发布。他们运营成本更低，迭代更快，改进自动复利。

坚守人类优先工作流的开发者会落后，不是因为他们是差劲的工程师，而是因为他们在优化错误的约束。

**先拿起螺丝刀，掌握基础，再升级到电锤。机车在等着。**
